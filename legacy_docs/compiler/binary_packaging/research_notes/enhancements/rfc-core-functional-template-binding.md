# RFC: Core Functional Template Binding for DFU3500 Probe Payloads

## Status

Proposed; revised after review.

The accepted thesis is unchanged: the next runtime milestone should be a tiny
functional payload generated by the current OpenFabric core pipeline under
`compiler/gpdpu_compiler/core`, not by the experimental `stream_compiler` branch
and not by the partner's original `testcase/application/*` / `build_app/run_mtr.sh`
workflow.

Reviewer feedback accepted in this revision:

- The first probe is fixed to `functional_maximum_single_app`, not log10.
- Runtime readiness is split into explicit manifest gates instead of one vague
  `functional_encoding` flag.
- Non-GEMM template rows should use `legacy_template_compat` or a predicate such
  as `uses_legacy_template_rows()`, rather than borrowing `legacy_gemm_compat`.
- Phase 2 uses fixed small templates and must fail closed; it is not a register
  allocator.
- Probe dtype/tolerance and normal load/store readiness must be explicit.

## Summary

`log10max_single_task` is now useful infrastructure, but it is not a functional
smoke.  It can generate package shape, runtime-control source, input data, and
reference metadata, but the device instruction rows are structural placeholders.
This is a good milestone: the current core pipeline is no longer blocked by
case staging or runtime-control assets.  It is blocked at the right target
boundary:

```text
TileMicroOp(local_compute / reduce_store / broadcast_load)
  -> Dfu3500TemplateBoundProgram
  -> ProgramBinRows functional instruction rows
```

The recommended next step is not to complete log10max immediately.  Instead,
add the smallest possible current-core functional payload:

```text
functional_maximum_single_app:
  Y = maximum(X, scalar_threshold)
```

This probe validates only the first executable non-GEMM local-compute slice:
input visibility, scalar immediate binding, `FMAX`, output store, runtime
control, and reference comparison.  Once this works, add affine arithmetic,
then log10, then finally return to reduce/app-storage/allreduce.

## Current State

Known facts:

- `log10max_single_task` is generated by `compiler/gpdpu_compiler/core`, outside
  `stream_compiler`.
- Runtime-control generation is payload-local and checked by
  `compiler/tools/check_runtime_control_generation.py`.
- `log10max_single_task` correctly reports itself as non-runnable:

```text
runtime_runnable=0
runtime_expectation=skip_functional_runtime
```

- The chip program status is:

```text
program_bin_package_structural_smoke_ready_functional_blocked
```

- `dfu3500_template_bound_program.validation` reports:

```text
all_micro_ops_have_template_bindings = false
unsupported_micro_op_count = 161
```

- Unsupported micro-op roles are currently:

```text
local_compute  : 144
broadcast_load : 16
reduce_store   : 1
```

- `program_bin_rows.inst_conf_report` reports:

```text
functional_encoding = false
functional_inst_row_count = 0
component_semantics_counts = { structural_smoke_only: 160 }
```

- Existing `legacy_gemm_compat` support binds GEMM-oriented micro-block kinds
  such as route materialization, accumulator prepare, compute update, and tile
  store.  It does not bind generic elementwise/reduce local compute.

Assumptions:

- The current normal single-app load/store path can either already produce real
  rows or can be made to do so with less uncertainty than collective/app-storage
  roles.  This must be verified during Phase 1; it must not be assumed silently.
- DFU3500 docs and vendor CSV evidence are sufficient to begin with simple
  lane-wise op templates such as `FMAX`, then `FMUL`, `FADD`, then `FLOG2`.
- The first functional probe should avoid `reduce_store`, `broadcast_load`,
  `app_storage_edge`, and multi-app handoff.
- A payload that runs runtime and fails output comparison is more valuable than
  a structural payload that only validates staging.

Open questions:

- Whether current normal input load/output store rows are already functional for
  the selected probe, or need an explicit minimal I/O binding step.
- Exact operand/register indices for fixed non-GEMM local-compute templates.
- Whether first probe should use fp16 I/O or another dtype already best supported
  by current runtime/output paths.
- Which component serializer gates must be complete before setting
  `runtime_runnable=1`.

Non-goals:

- Do not route this through `stream_compiler` yet.
- Do not use the partner original workflow as the proof path.
- Do not solve allreduce, app-storage materialization, broadcast load, or
  multi-app runtime sequencing in this phase.
- Do not make `log10max_single_task` pretend to be functional until reduce and
  storage semantics are truly bound.

## Problem

The current design fails as a functional probe because non-GEMM `local_compute`
roles have no executable DFU3500 template binding.  The core pipeline can
express log10max semantics:

```text
clamp_min -> log10 -> reduce_max -> threshold -> maximum -> affine -> store
```

but target binding only knows how to bind legacy GEMM-shaped roles.  Generic
compute becomes unsupported, and `ProgramBinRows` emits structural rows instead
of functional instruction encodings.

This is the correct failure mode.  The problem is now precise:

```text
OpenFabric core semantic/tile IR exists,
but DFU3500 target binding lacks executable rows for non-GEMM local compute.
```

The first fix must be small enough that failures are interpretable.  Starting
with full log10max would mix `local_compute`, `reduce_store`, `broadcast_load`,
allreduce/storage, serializer gates, runtime output collection, dtype conversion,
and numerical tolerance in one debugging step.

## Goals / Non-goals

Goals:

1. Add a tiny OpenFabric-owned current-core functional payload.
2. Make the first payload single-app and non-collective.
3. Bind `maximum_scalar` `local_compute` to real DFU3500 `LegacyInst` rows.
4. Keep local-compute template binding fixed and whitelisted; fail closed for
   unsupported compute kinds.
5. Split manifest/runtime gates so rows, package, runtime control, output
   collection, and reference checking are visible separately.
6. Preserve the guard that structural payloads are not runtime smokes.

Non-goals:

- Full log10max allreduce.
- General elementwise fusion.
- Generic register allocation.
- Partner original workflow parity.
- Stream compiler integration.
- Performance tuning.

## Proposed Design

### 1. Add `functional_maximum_single_app`

Add a new payload case under `build_payloads.py`:

```text
case_id=functional_maximum_single_app
formula=Y = maximum(X, scalar_threshold)
```

The case should generate:

```text
runtime/input_data.bin
runtime/riscv_src/riscv_control.json
runtime/riscv_src/riscv/testarm.c
reference/X.*.bin
reference/Y.*.bin
reference/reference.json
MANIFEST.txt
```

The first version remains blocked until row binding/package gates are complete:

```text
runtime_runnable=0
runtime_expectation=blocked_until_functional_rows_and_package_complete
```

It must not include:

```text
reduce_max
broadcast_load
reduce_store
app_storage_edge
multi-app handoff
```

### 2. Make load/store readiness explicit

Even the smallest probe still needs:

```text
input materialization / load
local_compute maximum_scalar
output store
```

Phase 1 must report whether normal single-app load/store rows are functional.
The manifest/debug plan should distinguish:

```text
load_rows_functional=true/false
local_compute_rows_functional=true/false
store_rows_functional=true/false
```

If load/store are not functional, the first implementation must bind the minimal
I/O envelope before claiming a runtime probe.  Binding only `FMAX` is not enough
for `runtime_runnable=1`.

### 3. Add `legacy_template_compat` or a row-mode predicate

Do not keep using `legacy_gemm_compat` as the name for non-GEMM functional
payloads.  Add either:

```text
vendor_inst_mode=legacy_template_compat
```

or an equivalent predicate:

```python
def uses_legacy_template_rows(vendor_inst_mode: str) -> bool:
    return vendor_inst_mode in {
        "legacy_gemm_compat",
        "legacy_template_compat",
    }
```

The bottom-level row infrastructure may be shared with GEMM compatibility, but
non-GEMM payloads must not communicate that they are GEMM.  Naming is part of
the boundary: future code must not infer GEMM semantics from the existence of
real legacy template rows.

### 4. Add a fixed small local-compute template binder

Add target binding for a minimal whitelist.  Phase 2 opens only one template:

```text
maximum_scalar
```

Conceptual fixed template:

```text
load/convert source lane value into temp0
bind scalar immediate threshold into temp1
FMAX temp0, temp1 -> dst
store/convert dst as required by output envelope
```

This binder is not a generic register allocator.  It should use fixed operand
slots/register roles for each supported compute kind.  If a `local_compute`
micro-op does not match the fixed supported shape, it must remain in
`unsupported_micro_ops` and the payload must remain non-runnable.

Future whitelist expansion:

```text
maximum_scalar      Phase 2
mul_scalar/add_scalar Phase 4
clamp_min/log10     Phase 5
```

### 5. Split manifest gates

`functional_encoding=true` is too coarse if read as runtime-ready.  Replace or
augment it with explicit gates:

```text
inst_rows_functional=true/false
runtime_package_complete=true/false
runtime_control_assets_valid=true/false
output_collection_supported=true/false
reference_check_available=true/false
runtime_runnable=true/false
```

Gate rule:

```text
runtime_runnable =
  inst_rows_functional
  && runtime_package_complete
  && runtime_control_assets_valid
  && output_collection_supported
  && reference_check_available
```

A state where rows are functional but component serialization is incomplete is
allowed, but it must not be described as runnable.

### 6. Fix dtype and tolerance per probe

For `functional_maximum_single_app`, avoid numerical complexity:

```text
input dtype    = choose current most stable runtime path
internal dtype = avoid conversion if possible
output dtype   = same as input unless current store path requires otherwise
comparison     = exact if dtype-stable, otherwise probe-local atol/rtol
```

Do not introduce `FLOG2` tolerance or conversion ambiguity in the first probe.
Those belong to later phases.

## Probe Acceptance Matrix

| Probe | Ops | Required roles | Required templates | Must avoid | Pass condition |
| --- | --- | --- | --- | --- | --- |
| `functional_maximum_single_app` | `maximum(X, scalar)` | normal load, local compute, normal store | scalar immediate, `FMAX`, I/O rows | reduce, broadcast, app-storage | runtime exits and output matches reference |
| `functional_affine_single_app` | `X * a + b` | normal load, local compute, normal store | scalar immediate, `FMUL`, `FADD`, I/O rows | reduce, broadcast, app-storage | runtime exits and output matches reference |
| `functional_log10_affine_single_app` | `log10(clamp_min(X)) * a + b` | normal load, local compute, normal store | `FMAX`, `FLOG2`, `FMUL`, `FADD`, conversion if needed | reduce, broadcast, app-storage | runtime exits and output matches tolerance |
| `log10max_single_task` | full log10max | local compute, reduce, storage, broadcast | all above plus reduce/storage/broadcast/allreduce support | pretending structural rows are functional | runtime exits and full reference check passes |

## Invariants

1. `ChipEnv` and frontend ops do not know DFU3500 register names, operand slots,
   or CSV row shapes.
2. `Dfu3500TemplateBoundProgram` owns target-template selection for tile
   micro-ops.
3. `ProgramBinRows` consumes template-bound rows; it must not rediscover compute
   semantics from chip ops.
4. A payload may set `runtime_runnable=1` only if all manifest gates are true.
5. `inst_rows_functional=true` requires every emitted row required by the probe
   to carry a real `LegacyInst`; no required row may be `structural_smoke_only`.
6. `unsupported_micro_op_count == 0` is required for the selected functional
   probe before runtime enablement.
7. Fixed local-compute templates fail closed.  Unsupported shapes must remain
   explicit diagnostics.
8. `log10max_single_task` remains blocked until local compute, reduce, storage,
   broadcast, component emission, runtime output collection, and reference check
   are all real.
9. The partner original workflow may be used as evidence, but not as the source
   of truth for OpenFabric core functional probe success.

## Alternatives Considered

### A. Start with log10 affine

Deferred.  It exercises `FLOG2`, constants, dtype conversion, and tolerance too
early.  It is a good third probe, not the first.

### B. Start with full log10max

Rejected for the next phase.  It mixes local compute, reduce, storage,
broadcast, allreduce, serializer gates, and runtime checking in one failure
surface.

### C. Derive from vendor `softmax_1`

Rejected as the primary proof path.  It proves that the vendor workflow can run,
not that OpenFabric's current core pipeline can generate a runnable operator
closure.

### D. Continue structural log10max probing

Rejected.  It validates package/control-plane plumbing, but cannot answer
whether device-side semantics are executable.

### E. Route the probe through `stream_compiler`

Deferred.  The B-line is promising, but using it now would make the first
functional runtime smoke depend on unfinished stream/fiber executable lowering.

## Migration / Implementation Plan

### Phase 1: Add report-only maximum probe

- Add `functional_maximum_single_app` to `build_payloads.py`.
- Generate deterministic input/reference/runtime-control metadata.
- Keep `runtime_runnable=0`.
- Report load/local-compute/store functional readiness separately.
- Confirm the probe has no reduce/app-storage/broadcast roles.

### Phase 2: Bind maximum scalar rows

- Add `legacy_template_compat` or `uses_legacy_template_rows()`.
- Add a fixed `maximum_scalar` local-compute binder.
- Validate `unsupported_micro_op_count == 0` for this probe.
- Validate every required instruction row has a real `LegacyInst`.
- Keep log10max blocked.

### Phase 3: Enable runtime only after package gates

- Set `runtime_runnable=1` only when:

```text
inst_rows_functional
runtime_package_complete
runtime_control_assets_valid
output_collection_supported
reference_check_available
```

- Upload and run on arch-13.
- Compare output against reference.

### Phase 4: Add affine probe

- Add `functional_affine_single_app`.
- Bind `mul_scalar` and `add_scalar` fixed templates.
- Run and compare.

### Phase 5: Add log10 affine probe

- Add `clamp_min` and `log10` via `FMAX`, `FLOG2`, `FMUL`, and any required
  conversion envelope.
- Define probe-local tolerance.
- Run and compare.

### Phase 6: Return to full log10max

- Bind or implement `reduce_store`, `broadcast_load`, and allreduce/app-storage
  semantics.
- Only then reconsider `log10max_single_task` as a functional runtime smoke.

## Risks and Mitigations

Risk: local compute binding grows into a hidden register allocator.

Mitigation: Phase 2 uses fixed templates only.  Unsupported shapes fail closed.

Risk: `legacy_gemm_compat` naming continues to leak GEMM assumptions into
non-GEMM probes.

Mitigation: add `legacy_template_compat` or a row-mode predicate before enabling
non-GEMM functional rows.

Risk: rows become functional while package/runtime remains incomplete.

Mitigation: split manifest gates and keep `runtime_runnable=0` until every gate
is true.

Risk: dtype conversion introduces ambiguity in the first probe.

Mitigation: use `maximum_scalar` first and choose the most stable existing dtype
path.  Defer FLOG2/conversion/tolerance to later probes.

Risk: load/store are assumed functional but are not.

Mitigation: Phase 1 must explicitly report load/store readiness.  If missing,
minimal I/O row binding becomes part of Phase 2 before runtime enablement.

## Expected Effect

After the first implementation phase, the project should have a tiny
OpenFabric-generated probe that honestly shows where current-core runtime
readiness stands.  After Phase 2/3, it should answer:

```text
Can current OpenFabric core emit real DFU3500 rows for a non-GEMM local compute
operator and run them under SimICT with reference checking?
```

This does not finish log10max.  It converts the current milestone from
"template binding is missing" into a precise sequence:

```text
maximum -> affine -> log10 -> log10max reduce/storage/broadcast
```

## Open Questions

1. Which dtype is safest for `functional_maximum_single_app` under current load
   and store paths?
2. Should the new row mode be a literal `legacy_template_compat` value, or only
   a predicate layered over existing modes?
3. Which normal load/store rows are already functional for non-GEMM payloads?
4. What is the minimal output collection and comparison hook for the first
   runtime pass?
5. Which component serializer gate must be completed before
   `runtime_package_complete=true`?

## Recommended Decision

Accept the direction with the stricter phase boundaries from this revision.
The next implementation should not use the partner original workflow and should
not touch `stream_compiler`.

Recommended immediate phase:

```text
Phase 1:
  Add functional_maximum_single_app as report-only.
  Generate input/reference/runtime-control metadata.
  Keep runtime_runnable=0.

Phase 2:
  Add legacy_template_compat or uses_legacy_template_rows().
  Bind only maximum_scalar local_compute first.
  Require unsupported_micro_op_count == 0 and real LegacyInst rows.

Phase 3:
  Enable runtime only when instruction rows, component package,
  runtime-control assets, output collection, and reference check are complete.
  Upload to arch-13 and compare output.
```
