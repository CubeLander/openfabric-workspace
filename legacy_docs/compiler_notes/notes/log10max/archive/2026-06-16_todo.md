# log10 -> max reduce -> maximum lowering TODO

This note tracks the non-binary work needed to make
`compiler/torch_examples/log10_maximum_dtensor.py` compile through the
OpenFabric/GPDPU lowering stack. Binary serialization is intentionally out of
scope here because another agent is working on it.

## Current boundary

- `compiler/torch_examples/log10_maximum_dtensor.py` is a PyTorch/DTensor
  semantic reference, not a compiler API entrypoint.
- The compiler frontend currently has first-class `matmul` and `relu` only.
- Generic PE-local actions can become `local_elementwise` tile phases, but
  `global_tile_dependency_network` used to collect only `local_gemm_summa`, so
  non-GEMM local phases disappeared before DFU graph construction.
- `route_lowering` already has a mesh visibility case for `all_reduce`, but it
  is still symbolic line-fanout, not a final reduce-tree or binary lowering.

## Implementation order

1. Add compiler frontend ops for the target expression:
   `clamp_min`, `log10`, `reduce_max`, `maximum`, and affine scalar helpers.
2. Add a real compiler example under `compiler/examples/` that mirrors the
   torch reference without depending on PyTorch.
3. Lower non-GEMM local phases into tile dependency ops so they survive into
   `dfu_graph`.
4. Add symbolic legacy-DFU assembly records for elementwise and local-reduce
   actions with `binary_status=unencoded`.
5. Keep collective max as a logical `all_reduce` route until the serializer and
   vendor reduce implementation are ready.
6. Add focused tests around `env.to_plan()` rather than `env.generate()` while
   binary/debug-dump WIP is moving.

## Expected phase shape

1. PE-local pre-reduce:
   `clamp_min(input, eps) -> log10 -> local_reduce_max`.
2. Mesh collective:
   `all_reduce(MAX)` over the per-PE scalar summary.
3. PE-local post-reduce:
   `global_max - margin -> maximum(log_spec, threshold) -> affine scale`.

## Review-defense notes

- This is not a GEMM-only abstraction: GEMM remains the first optimized
  specialization, while elementwise/reduce phases exercise the generic
  TileScope/TileOp path.
- The all-reduce is deliberately represented before binary lowering; this keeps
  cross-PE dependence explicit without pretending the current route pass already
  implements a production max-reduction tree.
- `log10` likely lowers to `FLOG2 * log10(2)` on DFU3500-class SIMD because the
  visible instruction surface has `FLOG2` but no direct `LOG10`.
